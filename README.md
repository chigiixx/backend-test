# Developer Landing Backend

Backend-сервис для лендинг-презентации разработчика: REST API формы обратной связи
с AI-анализом обращений, email-уведомлениями, rate limiting и файловым логированием
— плюс сам лендинг (`frontend/`), который реально отправляет запросы на этот API.

Выполнено как тестовое задание (Backend-ориентированное, Python/FastAPI вариант).

---

## 1. Как запустить проект

### Требования
- Python 3.9+ (разрабатывалось и тестировалось на 3.11/3.12)
- pip

### Установка и запуск локально

```bash
# 1. Клонировать репозиторий
git clone <ваш-репозиторий-url>
cd backend-test

# 2. Создать виртуальное окружение
python3 -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Настроить переменные окружения
cp .env.example .env
# откройте .env и при желании укажите OPENAI_API_KEY — без него сервис
# тоже полностью работает, просто AI-анализ переключится на fallback-режим

# 5. Запустить сервер
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Сервис будет доступен на `http://localhost:8000` — там сразу откроется
лендинг (`frontend/index.html`), форма на котором отправляет реальные
запросы на этот же backend. Swagger-документация — на
`http://localhost:8000/docs` (ReDoc — на `/redoc`), чистая информация
о сервисе в JSON — на `/api/info`.

### Запуск через Docker

```bash
docker build -t developer-landing-backend .
docker run -p 8000:8000 --env-file .env developer-landing-backend
```

### Запуск тестов

```bash
pytest -v
```

В тестовом окружении `OPENAI_API_KEY` намеренно не задаётся — это позволяет
проверить, что graceful fallback действительно работает и сервис не падает
без доступа к AI-провайдеру.

### Деплой

Локального запуска (см. выше) достаточно для проверки задания. Если нужен
рабочий публичный URL, проще всего задеплоить на **Render** (бесплатный tier).
Backend и фронтенд деплоятся вместе одним сервисом — FastAPI сам раздаёт
`frontend/` (см. раздел "Фронтенд" ниже), поэтому публичный URL сразу
открывает живой лендинг с рабочей формой:

1. Зарегистрироваться на render.com, New → Web Service → подключить GitHub-репозиторий.
2. Build command: `pip install -r requirements.txt`
3. Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
4. В Environment добавить переменные из `.env.example` (как минимум `EMAIL_BACKEND=mock`,
   опционально `OPENAI_API_KEY`).
5. После деплоя лендинг будет доступен по `https://<имя-сервиса>.onrender.com`,
   API — там же (`/api/...`), Swagger — по `/docs`.

Альтернатива — Railway, работает аналогично (Nixpacks сам распознает `requirements.txt`).

## 2. Стек технологий

**Backend:** Python 3.11+, FastAPI, Pydantic v2 / pydantic-settings (валидация и конфиг),
Uvicorn (ASGI-сервер).

**AI:** OpenAI API (`chat.completions`, модель `gpt-4o-mini` по умолчанию, настраивается
через `OPENAI_MODEL`) — анализ тональности и типа обращения. Если ключ не задан, библиотека
недоступна или провайдер вернул ошибку/таймаут — используется rule-based fallback на чистом
Python (без внешних вызовов).

**Хранение данных:** файловая система (JSON) — без БД, как допускает задание. Логи,
обращения, метрики и данные rate limiter'а хранятся в `data/`.

**Тестирование:** pytest, FastAPI `TestClient` (httpx).

**Инфраструктура:** `.env` для конфигурации, `Dockerfile` для контейнеризации.

## 3. Архитектура

Слоистая структура: **Controllers → Services → Repositories**, плюс отдельные
сквозные модули (`core`) для логирования, middleware и обработки ошибок.

```
app/
├── main.py                     # точка входа: FastAPI app, CORS, error handlers, роутеры
├── config.py                   # Settings (pydantic-settings), чтение .env
├── api/routes/                 # Controllers — тонкий слой HTTP
│   ├── contact.py              #   POST /api/contact
│   ├── health.py                #   GET  /api/health
│   └── metrics.py               #   GET  /api/metrics
├── schemas/contact.py          # Pydantic-схемы запросов/ответов + валидация
├── services/                   # Бизнес-логика
│   ├── contact_service.py      #   оркестрация: AI → email → сохранение → метрики
│   ├── ai_service.py           #   AI-анализ (OpenAI) + rule-based fallback
│   ├── email_service.py        #   отправка email (mock или smtp)
│   └── rate_limiter.py         #   файловый rate limiting
├── repositories/                # Доступ к данным (файловое хранилище)
│   ├── contact_repository.py    #   contacts.json
│   └── metrics_repository.py    #   metrics.json
└── core/
    ├── exceptions.py            # иерархия кастомных исключений (AppException → 4xx/5xx)
    ├── middleware.py            # логирование запросов + X-Request-ID
    └── logging_config.py        # настройка логгеров (консоль + ротация файлов)
```

**Почему так:**
- **Controllers** ничего не знают о хранении данных или о том, как реализован AI —
  они принимают HTTP-запрос, вызывают сервис и формируют ответ.
- **Services** содержат бизнес-логику и не зависят от деталей HTTP (их легко
  переиспользовать, например, в CLI-скрипте или фоновой задаче).
- **Repositories** изолируют способ хранения данных. Сейчас это JSON-файлы;
  чтобы перейти на PostgreSQL/MongoDB, нужно поменять только эти два класса —
  остальной код не затрагивается (зависимость от абстракции, а не от реализации).
- Кастомные исключения (`AppException` и наследники) дают предсказуемое
  отображение бизнес-ошибок в HTTP-статусы без `if/else` в контроллерах.

## 4. Реализация API

### `POST /api/contact`
Принимает обращение с формы обратной связи.

**Запрос:**
```json
{
  "name": "Иван Иванов",
  "phone": "+7 999 123-45-67",
  "email": "ivan@example.com",
  "comment": "Хочу обсудить разработку лендинга"
}
```

**Успешный ответ — `201 Created`:**
```json
{
  "id": "cf9bc2a8-0ffb-4e57-934e-16dc1703e9db",
  "status": "received",
  "message": "Спасибо! Ваше обращение получено, мы скоро с вами свяжемся.",
  "analysis": {
    "sentiment": "positive",
    "request_type": "feedback",
    "suggested_reply": "Спасибо за ваше обращение! ...",
    "ai_used": false
  },
  "created_at": "2026-06-19T18:23:11.730450+00:00"
}
```

**Ошибка валидации — `422 Unprocessable Entity`:**
```json
{
  "error": "validation_error",
  "detail": [
    {"loc": ["body", "email"], "msg": "value is not a valid email address...", "type": "value_error"}
  ],
  "request_id": "c6639ed4-..."
}
```

**Превышен rate limit — `429 Too Many Requests`:**
```json
{
  "error": "rate_limit_exceeded",
  "detail": "Слишком много запросов. Попробуйте снова через 593 сек.",
  "request_id": "922eb4d3-..."
}
```

### `GET /api/health`
Статус сервиса: запущен ли, сконфигурирован ли AI-провайдер, какой backend
у email, аптайм. Полезно для мониторинга/healthcheck'ов при деплое.

### `GET /api/metrics`
Агрегированная статистика: количество обращений, разбивка по тональности и типу
запроса, сколько раз сработал реальный AI vs fallback, сколько запросов отклонено
rate limiter'ом.

Полный список эндпоинтов, схемы и возможность "попробовать" запрос —
в Swagger UI (`/docs`) и ReDoc (`/redoc`), генерируется автоматически из кода
(FastAPI + Pydantic). Примеры curl/Postman — см. `postman_collection.json`
и команды ниже.

```bash
curl -X POST http://localhost:8000/api/contact \
  -H "Content-Type: application/json" \
  -d '{"name":"Иван Иванов","phone":"+7 999 123-45-67","email":"ivan@example.com","comment":"Отличный сайт!"}'

curl http://localhost:8000/api/health
curl http://localhost:8000/api/metrics
```

### Валидация и обработка ошибок
- **Валидация** — декларативно на уровне Pydantic-схемы (`app/schemas/contact.py`):
  длина имени, формат и длина телефона (regex + проверка количества цифр),
  корректность email (`EmailStr`), длина комментария.
- **Глобальный error handler** (`app/main.py`) перехватывает три уровня ошибок:
  1. `AppException` и наследники (бизнес-ошибки: rate limit и т.д.) → нужный статус-код.
  2. `RequestValidationError` (ошибки Pydantic) → `422` с детальным списком проблем.
  3. Любое необработанное исключение → `500` с общим сообщением (детали — только в логах,
     наружу не утекают).
- Каждый ответ об ошибке содержит `request_id`, который также пишется в лог —
  удобно сопоставить жалобу пользователя с конкретной записью в `app.log`.
- **CORS** настраивается через `CORS_ORIGINS` в `.env` (по умолчанию `*` для удобства
  локальной разработки; на проде стоит указать конкретный домен лендинга).

## 5. AI-интеграция

**Провайдер:** OpenAI API, эндпоинт `chat.completions` с `response_format: json_object`
(модель отвечает строго JSON-объектом, без парсинга текста регулярками).

**Что делает AI** (`app/services/ai_service.py`): по тексту комментария определяет
- `sentiment` — тональность (`positive` / `neutral` / `negative`),
- `request_type` — тип обращения (`question` / `complaint` / `feedback` / `proposal` / `other`),
- `suggested_reply` — короткий черновик ответа пользователю на русском языке.

**Системный промпт:**
```
Ты — ассистент, который анализирует обращения с формы обратной связи на сайте
разработчика-фрилансера. Проанализируй комментарий пользователя и верни СТРОГО
валидный JSON без какого-либо текста до или после, в формате:
{"sentiment": "positive"|"neutral"|"negative",
 "request_type": "question"|"complaint"|"feedback"|"proposal"|"other",
 "suggested_reply": "короткий вежливый черновик ответа на русском языке, 1-2 предложения"}
```
В `messages` дальше передаётся комментарий пользователя как `user`-сообщение.

**Graceful fallback.** Это ключевое требование задания, и оно реализовано на
нескольких уровнях:
- Если `OPENAI_API_KEY` не задан в `.env` — AI-клиент вообще не создаётся,
  сразу используется fallback (без лишних попыток обращения к сети).
- Если ключ задан, но запрос упал по любой причине (таймаут — ограничен
  `AI_TIMEOUT_SECONDS`, недоступность сети, лимиты провайдера, невалидный JSON
  в ответе) — исключение перехватывается, пишется `WARNING` в лог, и сервис
  переключается на fallback **в рамках того же запроса**, не возвращая ошибку
  клиенту.
- **Fallback** — простой rule-based анализ: списки позитивных/негативных слов
  (RU/EN) для тональности, эвристики (наличие `?`, тон) для типа обращения,
  и заранее заготовленный вежливый шаблон ответа.
- В ответе API всегда есть поле `analysis.ai_used: true/false` — честно показывает,
  использовался ли реальный AI или fallback, это же видно в `/api/metrics`
  (`ai_used_count` vs `ai_fallback_count`).

Таким образом обращение **никогда** не теряется и не падает с ошибкой из-за
проблем с AI-провайдером — деградация плавная, а не отказ сервиса.

## 6. Что сделано с помощью AI

Честно: весь код этого проекта (структура, бэкенд-логика, тесты, README) был
сгенерирован в паре с Claude (Anthropic) на основе текста тестового задания —
я не скрываю этого, так как раздел задания явно просит это описать.

**Что генерировалось AI:**
- Вся структура проекта и слоистая архитектура (Controllers/Services/Repositories).
- Реализация всех эндпоинтов, валидации, rate limiter'а, email-сервиса,
  AI-сервиса с fallback, глобальной обработки ошибок и логирования.
- Тесты (`tests/test_contact.py`), Postman-коллекция, Dockerfile, README.
- Фронтенд-лендинг (`frontend/`) — HTML/CSS/JS без фреймворков, подключённый
  к реальному API (см. раздел «Фронтенд» ниже).

**Что было проверено и исправлено вручную в процессе:**
- Запущен реальный сервер (`uvicorn`) и прогнаны live-запросы через `curl`
  для всех эндпоинтов (успех, валидационные ошибки, rate limit, AI-fallback) —
  не только юнит-тесты "на бумаге".
- Найден и исправлен реальный баг: `RequestValidationError.errors()` от Pydantic
  кладёт в `ctx` сырой объект исключения (`ValueError`) из кастомных
  `@field_validator`, из-за чего `JSONResponse` падал с `TypeError: Object of
  type ValueError is not JSON serializable` при невалидном телефоне. Исправлено
  явной сборкой безопасного для сериализации списка ошибок (`loc`/`msg`/`type`).
- Заменён устаревший `@app.on_event("startup")` на современный `lifespan`
  context manager (FastAPI выдавал `DeprecationWarning`).
- При добавлении фронтенда проверено (тестами и вручную через `curl`), что
  `app.mount("/", StaticFiles(...))` не перехватывает запросы к `/api/*`,
  `/docs`, `/redoc`, `/openapi.json` — порядок регистрации роутов в FastAPI
  имеет значение, и это нужно было явно проверить, а не предполагать.
- Все 10 тестов и весь happy-path/edge-case флоу прогнаны вручную перед сдачей.

**Что стоит сделать перед собеседованием (рекомендация себе):**
прочитать каждый файл и быть готовым объяснить любое решение своими словами —
особенно почему слоистая архитектура, почему файловое хранилище, как именно
работает fallback и rate limiter. Само по себе использование AI для генерации
кода — это и есть то, что просит само задание ("используете AI-инструменты в
backend" и "что написать в README" → "что сделано с помощью AI"), но способность
аргументированно объяснить решения на интервью важнее факта, что код написан с
помощью AI.

## 7. Хранение данных

База данных не используется — всё хранится в JSON-файлах в каталоге `data/`
(каталог создаётся автоматически при первом запуске, если его нет):

```
data/
├── logs/
│   ├── app.log         # все логи приложения (с ротацией, 5MB × 3 файла)
│   └── access.log      # лог каждого HTTP-запроса: IP, метод, путь, статус, время, request_id
├── storage/
│   ├── contacts.json   # все полученные обращения (со временем, AI-анализом, статусом email)
│   ├── metrics.json    # агрегированная статистика для GET /api/metrics
│   └── rate_limit.json # временные метки запросов по IP для rate limiter'а
└── emails/
    └── *.txt            # "отправленные" письма в mock-режиме (см. ниже)
```

**Логи** (`app/core/logging_config.py`): `RotatingFileHandler` на 5 МБ с 3 резервными
файлами + дублирование в консоль. Отдельный логгер `access` пишет одну строку на
каждый HTTP-запрос (через middleware в `app/core/middleware.py`).

**Rate limiting** (`app/services/rate_limiter.py`): для каждого IP в `rate_limit.json`
хранится список временных меток последних запросов (sliding window). При новом
запросе старые метки за пределами окна (`RATE_LIMIT_WINDOW_SECONDS`, по умолчанию
600 сек) отбрасываются; если оставшихся меток `>= RATE_LIMIT_MAX_REQUESTS`
(по умолчанию 5) — запрос отклоняется с `429`. Доступ к файлу защищён
`threading.Lock` от гонок при параллельных запросах в одном процессе.

**Email (mock-режим)** (`app/services/email_service.py`, `EMAIL_BACKEND=mock` по
умолчанию): вместо реальной отправки письмо сохраняется как `.txt`-файл в
`data/emails/` (с темой, получателем и телом письма) и пишется в лог — так
проверяющий может убедиться, что письма владельцу и пользователю формируются
корректно, без необходимости настраивать реальный SMTP. При желании показать
реальную отправку — достаточно поставить `EMAIL_BACKEND=smtp` и заполнить
`SMTP_*` переменные в `.env`, код для этого уже реализован (`smtplib`,
`_send_smtp` в том же сервисе) — переключение бэкенда не требует правок в
остальном приложении.

**Статистика** (`metrics.json`) обновляется инкрементально при каждом обращении
(не пересчитывается из `contacts.json` на каждый запрос к `/api/metrics`),
поэтому эндпоинт остаётся быстрым независимо от объёма накопленных данных.

## Фронтенд

`frontend/` — статический лендинг (чистые HTML/CSS/JS, без сборки и
фреймворков), который FastAPI раздаёт сам через `StaticFiles` (см. конец
`app/main.py`): один деплой — и API, и презентационная страница работают
по одному адресу.

**Идея дизайна.** Страница оформлена в логике самого API-сервиса:
навигация — это список "роутов" (`GET /about`, `GET /skills`, `POST /contact`)
с HTTP-методами вместо иконок, метки полей формы совпадают с реальными
именами JSON-полей (`name`, `phone`, `email`, `comment`), которые уходят
в `POST /api/contact`. Эта метафора не просто оформление: она отражает
то, чем разработчик реально занимается.

**Что по-настоящему работает (не моки):**
- Hero-блок при загрузке страницы делает живой `GET /api/health` и
  печатает реальный ответ сервера (с аккуратным fallback-сообщением,
  если backend не запущен).
- Футер показывает статус API (онлайн/офлайн) через тот же health-check.
- Форма обратной связи отправляет настоящий `POST /api/contact`: при
  успехе показывает реальный AI-анализ (тональность, тип обращения,
  черновик ответа), при ошибке — честно показывает ошибки валидации
  под нужными полями или сообщение о rate limit, как их вернул backend.

**Контент страницы взят из настоящего резюме** (имя, реальный опыт работы,
стек, проекты — itrstroy.ru, стажировки и т.д.) — это уже не плейсхолдеры.
Если что-то поменяется (новое место работы, новые проекты) — правьте прямо
в `frontend/index.html`, разметка простая и без сборки.

**Если фронтенд нужно хостить отдельно** от backend (например, GitHub Pages,
а API — на Render): откройте `frontend/js/script.js` и укажите в начале файла
`API_BASE_URL = 'https://ваш-api.onrender.com'` вместо пустой строки. CORS уже
настроен (`CORS_ORIGINS` в `.env`) — достаточно прописать туда домен фронтенда.
